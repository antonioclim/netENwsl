Problema Complexitatii

De ce avem nevoie of modele arhitecturof in retelistica

Provocarea: Hardware divers, software variat, protocols multiple‚Äîtoate trebuind to coopereze global, fabricate of diferiti furnizori, at decenii away.

Far Layerificare

  • Fiecare aplicatie must to inteleag each tehnologie of network
  • Schimbarea hardware-ului inseamn rewritesrea software-ului
  • Far interoperabilitate between furnizori

Rezultat: Haos. Insule proprietare.

Cu Layerificare

  • Fiecare nivel are responsabilitati focalizate
  • Nivelurile interactioneaz through interfaces bine definite
  • Componentele pot fi schimbate far a afecta altele

Rezultat: Internetul global. Trilioane of dispozitive.

Analogia Constructiei

Aderstandrea arhitecturii layerede through constructii

Separarea Responsabilitatilor: Electricianul not modific fundatia. Designerul of interior not reinstalls cablajul.

  • Fiecare specialist lucreaz at propriul nivel
  • Nivelurile superioare se bazeaz pe servicesle nivelurilor inferioare
  • Schimbarile at un nivel not requires reconstruirea alttime

In retelistica: Dezvoltatorul web writes HTTP far to inteleag fibra optica.

Modelul of Referint OSI

OSI is un model of referinta. TCP/IP is model of implementare.

Nivelurile 1 & 2 ‚Äì Fizic and Data Link

Fundamentul comunicatiei in network

Nivelul 1 ‚Äì Fizic

  • Functie: Transmite bits brut through mediul fizic
  • PDU: Bit (sau simbol)
  • Preocupari: Niveluri of tensiune, sincronizare, conectori
  • Exemple: Cabluri Ethernet, optical fibre, radio Wi-Fi

Standard cheie: IEEE 802.3 (physical layer Ethernet)

Nivelul 2 ‚Äì Data Link

  • Functie: Livrare node-la-node, detectia errorslor
  • PDU: Cadru
  • Adresare: Adrese MAC (48 bits)
  • Exemple: Ethernet, Wi-Fi, PPP

MAC Address: 00:1A:2B:3C:4D:5E ‚Äì identificator hardware

Nivelul 3 ‚Äì Network

Adresarea logic and routinga between networks

Responsabilitati Principof

  • Adresare logica: Adresele IP identific global gazdele
  • Rutare: Determinarea pathsi through networks
  • Fragmentare: Impartirea packageelor for MTU-uri mhours smalli

MAC vs IP: Diferenta Critica

MAC (Nivelul 2)IP (Nivelul 3)
Fizic (hardware)Logic (configurabil)
Domeniu local (aceeasi LAN)Domeniu global (rutabil)
Se schimb at each hopRamane constant cap-la-cap

La each hop of router, addressesle MAC se schimba, but addressesle IP raman aceleasi!

Nivelul 4 ‚Äì Transport

Comunicare cap-la-cap between processes

TCP ‚Äì Transmission Control Protocol

  • Orientat conexiune: handshake in 3 pasi
  • Fiabil: Confirmari, retransmisii
  • Ordonat: Datele are livrate in secventa
  • Control flow & congestion: Incorptimet

Folositi cand: Fiecare byte matters (HTTP, SSH, email)

UDP ‚Äì User Datagram Protocol

  • Far conexiune: Far handshake, far stare
  • Nefiabil: Far garantii
  • Neordonat: Pachetele pot ajunge in alt ordine
  • Overhead minim: Header of 8 bytes

Folositi cand: Viteza matters (DNS, VoIP, gaming)

Socket: Combinatia (IP address, numar port) identific unic un punct final. Numerele of port are 0‚Äì65535.

Nivelurile Superioare (5-6-7)

Sesiune, Prezentare and Aplicatie

Nivelul 5 ‚Äì Sesiune

Gestioneaz dialogul between applications: stabilire, mentinere, incheiere.

Exemple: NetBIOS, RPC

Nivelul 6 ‚Äì Prezentare

Traducerea datalor, criptare, compresie for interpretare uniforma.

Exemple: TLS/SSL, JPEG, UTF-8

Nivelul 7 ‚Äì Aplicatie

Protocoof and services orientate utilizator. Interfata with which interactioneaz oamenii.

Exemple: HTTP, FTP, SMTP, DNS

In TCP/IP, nivelurile 5‚Äì7 are combinate intr-un singur nivel ‚ÄûAplicatie". Majoritatea protocolslor moderne gestioneaz aceste preocupari intern.

Incapsularea

Cum are impackageate the data at each nivel

Incapsulare: Procesul of impackageare a datalor with headere of protocol (si sometimes trailere) on masur ce trec in jos through stiva of network.

Header-ul fiecarui nivel contains: Informatii of control for the protocol acelui nivel. Payload-ul of deasupra becomes data opace.

Introduction in API-ul Socket

Questiona: Aderstandm nivelurile‚Äîdar how writesm programs which comunic through networks?

API-ul Socket

Socket-urile offers interface of progralarge between the application dvs. (Nivelul 7) and transport layer (Nivelul 4).

Socket: A punct final for comunicare. Identificat through (IP address, numar port). API-ul allows crearea connectionslor, sendsrea and receptia datalor.

Historical Note

API-ul socket a fost introdus in BSD UNIX 4.2 (1983). A became standardul of facto for progralargea of network.

Acelasi API is folosit in aproape orice SO: Linux, Windows (Winsock), macOS.

Tipuri of Socket

Alegerea socket-ului potrivit for nevoile dvs.

SOCK_STREAM (TCP)

  • Orientat conexiune
  • Flux of bytes fiabil, ordonat
  • Necesit connect() befhours of sendsre
  • Use send() / recv()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(("example.com", 80))

SOCK_DGRAM (UDP)

  • Far conexiune
  • Nefiabil, orientat messages
  • Nu requires conexiune
  • Use sendto() / recvfrom()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.sendto(data, ("example.com", 53))

Example Server TCP in Python

Din starterkit: ex_2_01_tcp.py

import socket

# Create socket: AF_INET = IPv4, SOCK_STREAM = TCP
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
sock.bind(("0.0.0.0", 9090))  # Leag on all interfacesle
sock.listen(32)  # Backlog of 32 connections in asteptare

while True:
    conn, addr = sock.accept()  # Block pan se connects un client
    data = conn.recv(1024)
    response = b"OK: " + data.upper()
    conn.sendall(response)
    conn.close()

Conceptii Gresite Frecvente

Clarificarea confuziilor comune

‚ùå Conceptie Gresita: ‚ÄûDatele calathourssc orizontal between niveluri pereche."

‚úì Realitate: Datele calathourssc in jos through stiv at emitator, through mediul fizic, then in sus at receptor.

‚ùå Conceptie Gresita: ‚ÄûTCP garanteaz livrarea."

‚úì Realitate: TCP garanteaz livrarea sau notificarea esecului. If the network e inaccesibila, TCP renunt in final.

‚ùå Conceptie Gresita: ‚ÄûOSI Model is folosit in networks reof."

‚úì Realitate: TCP/IP is ce e implementat. OSI is un model of referint for discutii.

‚ùå Conceptie Gresita: ‚ÄûAdresele MAC not se schimb at routing."

‚úì Realitate: MAC Sursa/Destinatie se schimb at each hop. Adresele IP raman constante (cu exceptia NAT).
