The Protocol Wars: A Cautionary Tale

How pragmatism defeated bureaucracy

In the 1980s, the telecommunications industry backed the OSI model‚Äîcommittee-designed with volumes of documentation. Meanwhile, the academic community championed TCP/IP with "rough consensus and running code."

OSI collapsed under its complexity. TCP/IP, bundled with BSD UNIX from 1983, triumphed. Universities initially planned to use "TCP/IP until OSI becomes available"‚Äîthen quietly dropped OSI altogether.

OSI's Downfall

  • Seven years from design to first implementation
  • Specification documents measured in kilograms
  • Design by committee, not engineers

TCP/IP's Victory

  • Free reference implementation in BSD UNIX
  • Tested at scale before standardisation
  • "The code is the specification"

The Problem of Complexity

Why we need architectural models in networking

The Challenge: Diverse hardware, varied software, multiple protocols‚Äîall needing to cooperate globally, manufactured by different vendors, decades apart.

Without Layering

  • Every application must understand every network technology
  • Changing hardware means rewriting software
  • No interoperability between vendors

Result: Chaos. Proprietary islands.

With Layering

  • Each layer has focused responsibilities
  • Layers interact through well-defined interfaces
  • Components can be swapped without affecting others

Result: The global Internet. Trillions of devices.

The Building Analogy

Understanding layered architecture through construction

Separation of Concerns: The electrician doesn't modify the foundation. The interior designer doesn't rewire the building.

  • Each specialist works at their own layer
  • Upper layers rely on services from lower layers
  • Changes in one layer don't require rebuilding others

In networking: The web developer writes HTTP without understanding fibre optics.

The Layer Concept

How layers communicate with each other

Fundamental Principles

  • A layer provides services to the layer above
  • A layer uses services from the layer below
  • Each layer only "sees" adjacent layers
  • Horizontal communication is virtual‚Äîdata flows vertically

Service Access Point (SAP): The interface through which a layer provides services to the layer above.

Key Terminology

TermMeaning
PDUProtocol Data Unit ‚Äì data at any given layer
SDUService Data Unit ‚Äì payload from layer above

TrailerControl info at end (typically L2)

The OSI Reference Model

Open Systems Interconnection ‚Äì The theoretical framework

OSI = Open Systems Interconnection

Standardised by ISO in 1984 (ISO 7498)

Characteristics

  • Theoretical reference model
  • Seven distinct layers
  • Developed as response to proprietary systems
  • Implementation-independent specification

Practical Purpose Today

OSI provides a common vocabulary for discussing networks. Even though TCP/IP is used in practice, we describe problems using OSI terminology.

OSI is a reference model. TCP/IP is the implementation model.

Layers 1 & 2 ‚Äì Physical and Data Link

The foundation of network communication

Layer 1 ‚Äì Physical

  • Function: Transmit raw bits over physical medium
  • PDU: Bit (or symbol)
  • Concerns: Voltage levels, timing, connectors
  • Examples: Ethernet cables, fibre optics, Wi-Fi radio

Key standard: IEEE 802.3 (Ethernet physical layer)

Layer 2 ‚Äì Data Link

  • Function: Node-to-node delivery, error detection
  • PDU: Frame
  • Addressing: MAC addresses (48-bit)
  • Examples: Ethernet, Wi-Fi, PPP

MAC Address: 00:1A:2B:3C:4D:5E ‚Äì hardware identifier

Layer 3 ‚Äì Network

Logical addressing and routing between networks

AspectDetails
FunctionRoute packets between networks
PDUPacket
AddressingIP addresses (32-bit v4, 128-bit v6)
ProtocolsIP, ICMP, ARP, routing protocols

Core Responsibilities

  • Logical addressing: IP addresses identify hosts globally
  • Routing: Determining path across networks
  • Fragmentation: Breaking packets for smaller MTUs

MAC vs IP: Critical Difference

MAC (Layer 2)IP (Layer 3)
Physical (hardware)Logical (configurable)
Local scope (same LAN)Global scope (routable)
Changes at each hopRemains constant end-to-end

At each router hop, MAC addresses change, but IP addresses remain the same!

Layer 4 ‚Äì Transport

End-to-end communication between processes

TCP ‚Äì Transmission Control Protocol

  • Connection-oriented: 3-way handshake
  • Reliable: Acknowledgements, retransmissions
  • Ordered: Data delivered in sequence
  • Flow & congestion control: Built-in

Use when: Every byte matters (HTTP, SSH, email)

UDP ‚Äì User Datagram Protocol

  • Connectionless: No handshake, no state
  • Unreliable: No guarantees
  • Unordered: Packets may arrive out of sequence
  • Minimal overhead: 8-byte header

Use when: Speed matters (DNS, VoIP, gaming)

Socket: The combination (IP address, Port number) uniquely identifies an endpoint. Port numbers range 0‚Äì65535.

Upper Layers (5-6-7)

Session, Presentation, and Application

Layer 5 ‚Äì Session

Manages dialogue between applications: establishment, maintenance, termination.

Examples: NetBIOS, RPC

Layer 6 ‚Äì Presentation

Data translation, encryption, compression for uniform interpretation.

Examples: TLS/SSL, JPEG, UTF-8

Layer 7 ‚Äì Application

User-facing protocols and services. The interface humans interact with.

Examples: HTTP, FTP, SMTP, DNS

In TCP/IP, layers 5‚Äì7 are combined into a single "Application" layer. Most modern protocols handle these concerns internally.

Knowledge Check

Test your understanding of OSI layers

Question: What type of addressing does Layer 2 (Data Link) use?

Correct! MAC (Media Access Control) addresses are 48-bit hardware identifiers used at Layer 2 for local network communication. IP addresses operate at Layer 3, port numbers at Layer 4.

Encapsulation

How data is wrapped at each layer

Encapsulation: The process of wrapping data with protocol headers (and sometimes trailers) as it passes down through the network stack.

Each layer's header contains: Control information for that layer's protocol. The payload from above becomes opaque data.

The TCP/IP Model

The practical implementation model

Application Layer

HTTP, FTP, SMTP, DNS, SSH, DHCP

Combines OSI L5-6-7

Transport Layer

TCP, UDP, SCTP

= OSI L4

Internet Layer

IP (v4/v6), ICMP

= OSI L3

Network Access Layer

Ethernet, Wi-Fi, PPP

Combines OSI L1-2

TCP/IP = Transmission Control Protocol / Internet Protocol

Developed by DARPA (1970s), standardised through RFCs

Why TCP/IP Won

  • Working code before complete specification
  • Free reference implementation in BSD UNIX
  • Tested at scale on ARPANET
  • "Rough consensus and running code"

OSI vs TCP/IP Comparison

Key differences between the models

AspectOSI ModelTCP/IP Model

OSI Advantage: Clearer separation of concerns, better for teaching.

TCP/IP Advantage: Reflects actual implementations, simpler.

Knowledge Check

TCP/IP Model understanding

Question: In which order are headers added when sending an HTTP request?

Correct! Data flows down the stack. HTTP data is generated at application layer, then encapsulated with TCP header (transport), IP header (internet), and finally Ethernet header/trailer (network access).

From Theory to Code

Introduction to the Socket API

The Question: We understand the layers‚Äîbut how do we actually write programs that communicate over networks?

The Socket API

Sockets provide the programming interface between your application (Layer 7) and the transport layer (Layer 4).

Socket: An endpoint for communication. Identified by (IP address, port number). The API allows creating connections, sending data, and receiving data.

Historical Note

The socket API was introduced in BSD UNIX 4.2 (1983). It became the de facto standard for network programming.

The same API is used in virtually every OS: Linux, Windows (Winsock), macOS.

Socket Types

Choosing the right socket for your needs

SOCK_STREAM (TCP)

  • Connection-oriented
  • Reliable, ordered byte stream
  • Requires connect() before sending
  • Use send() / recv()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(("example.com", 80))

SOCK_DGRAM (UDP)

  • Connectionless
  • Unreliable, message-oriented
  • No connection needed
  • Use sendto() / recvfrom()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.sendto(data, ("example.com", 53))

Python TCP Server Example

From the starterkit: ex_2_01_tcp.py

import socket

# Create socket: AF_INET = IPv4, SOCK_STREAM = TCP
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
sock.bind(("0.0.0.0", 9090))  # Bind to all interfaces
sock.listen(32)  # Backlog of 32 pending connections

while True:
    conn, addr = sock.accept()  # Blocks until client connects
    data = conn.recv(1024)
    response = b"OK: " + data.upper()
    conn.sendall(response)
    conn.close()

Try it: python3 seminar/python/exercises/ex_2_01_tcp.py server --port 9090

TCP Three-Way Handshake

Establishing a reliable connection

Why three messages? Each side must confirm it can both send and receive. SYN establishes client‚Üíserver. SYN-ACK confirms server received. Final ACK confirms client received.

Common Misconceptions

Clearing up frequent confusions

‚ùå Misconception: "Data travels horizontally between peer layers."

‚úì Reality: Data travels down the stack on sender, across physical medium, then up on receiver.

‚ùå Misconception: "TCP guarantees delivery."

‚úì Reality: TCP guarantees delivery or notification of failure. If network unreachable, TCP eventually gives up.

‚ùå Misconception: "The OSI model is used in real networks."

‚úì Reality: TCP/IP is what's implemented. OSI is a reference model for discussion.

‚ùå Misconception: "MAC addresses don't change during routing."

‚úì Reality: Source/Dest MAC change at each hop. IP addresses remain constant (except NAT).
